# Data Model Specification

## Purpose

ncps uses a relational database for all metadata. The database engine is selected at runtime; SQL queries are generated by sqlc; migrations are managed by dbmate. This specification captures the database engines supported, the toolchain conventions, the schema of every table, and the public `database.Querier` surface exposed to the cache layer.

## Requirements

### Requirement: Engine selection by URL scheme

The system SHALL select the database engine at runtime from the scheme of the `--cache-database-url` flag. The function `database.Open(dbURL, poolCfg)` SHALL return a `database.Querier` interface implementation backed by the appropriate engine wrapper.

#### Scenario: Selecting SQLite

- **WHEN** `--cache-database-url=sqlite:/path/to/db` is passed
- **THEN** `database.Open` SHALL return a `Querier` backed by the `pkg/database/sqlitedb` adapter
- **AND** the SQLite driver SHALL be configured with `MaxOpenConns=1`, WAL mode, `busy_timeout=10000ms`, and `foreign_keys=ON`

#### Scenario: Selecting PostgreSQL

- **WHEN** `--cache-database-url=postgresql://...` or `--cache-database-url=postgres://...` is passed
- **THEN** `database.Open` SHALL return a `Querier` backed by the `pkg/database/postgresdb` adapter
- **AND** the PostgreSQL driver pool SHALL be configurable via `PoolConfig`

#### Scenario: Selecting MySQL/MariaDB

- **WHEN** `--cache-database-url=mysql://...` or `--cache-database-url=mysql+unix://...` is passed
- **THEN** `database.Open` SHALL return a `Querier` backed by the `pkg/database/mysqldb` adapter
- **AND** the MySQL driver SHALL be configured with `parseTime=true`, `loc=UTC`, and the connection SHALL force `time_zone='+00:00'`

#### Scenario: Unsupported scheme

- **WHEN** a URL whose scheme is not one of the above is passed
- **THEN** `database.Open` SHALL return an error of type `database.ErrUnsupportedDriver`

### Requirement: sqlc generates per-engine Querier interfaces

The system SHALL use sqlc (configured via `sqlc.yaml` at the repository root, with `version: "2"`) to generate three completely separate `Querier` interfaces — one per engine package (`sqlitedb.Querier`, `postgresdb.Querier`, `mysqldb.Querier`). The three interfaces SHALL NOT be mixed at runtime; each engine SHALL exclusively use its own generated interface.

#### Scenario: Three sqlc entries

- **WHEN** a developer inspects `sqlc.yaml`
- **THEN** the file SHALL declare three `sql` entries — `sqlite`, `postgresql`, and `mysql` — each pointing at its own query file under `db/query.<engine>.sql` and its own output package under `pkg/database/<engine>db/`

#### Scenario: emit settings

- **WHEN** sqlc is run
- **THEN** each generated package SHALL include the SQL as a method comment (`emit_sql_as_comment: true`) and SHALL expose a `Querier` interface (`emit_interface: true`)

#### Scenario: No cross-engine type pollution

- **WHEN** application code consumes a `Querier`
- **THEN** the concrete type SHALL be the engine-specific generated wrapper, and SHALL NOT accept entities from a different engine's package

### Requirement: Hand-written `database.Querier` superset interface

The system SHALL expose a hand-written `database.Querier` interface in `pkg/database/` that is a superset of every method on the three generated per-engine interfaces. Each engine-specific wrapper (`sqliteWrapper`, `postgresWrapper`, `mysqlWrapper`) SHALL implement this hand-written interface by delegating to its sqlc-generated adapter.

#### Scenario: Caller imports only `database`

- **WHEN** code in `pkg/cache/` declares a dependency on the database
- **THEN** it SHALL import only `github.com/kalbasit/ncps/pkg/database` and use the `database.Querier` interface
- **AND** it SHALL NOT import any engine-specific subpackage

### Requirement: sqlc type and column overrides

The system SHALL declare the following type overrides in `sqlc.yaml`: `nar_files.file_size → uint64`, `chunks.size → uint32`, `chunks.compressed_size → uint32`, and `nar_files.chunking_started_at → database/sql.NullTime`. The system SHALL also declare the column renames `narinfo_id → NarInfoID` and `url → URL` so that generated Go fields use idiomatic acronym capitalization.

#### Scenario: Generated NarFile has uint64 file_size

- **WHEN** sqlc generates the `NarFile` struct
- **THEN** the `FileSize` field SHALL have type `uint64`

#### Scenario: Generated Chunk uses uint32 sizes

- **WHEN** sqlc generates the `Chunk` struct
- **THEN** both `Size` and `CompressedSize` fields SHALL have type `uint32`

#### Scenario: Generated NarFile has nullable chunking_started_at

- **WHEN** sqlc generates the `NarFile` struct
- **THEN** the `ChunkingStartedAt` field SHALL have type `database/sql.NullTime`

#### Scenario: Acronym capitalization

- **WHEN** sqlc generates structs and parameter types referencing `narinfo_id` or `url`
- **THEN** the Go identifiers SHALL be `NarInfoID` and `URL` respectively

### Requirement: sqlc regeneration workflow

The system SHALL require running `sqlc generate` followed by `go generate ./pkg/database` after any modification to `db/query.sqlite.sql`, `db/query.postgres.sql`, or `db/query.mysql.sql`.

#### Scenario: Updating a query

- **WHEN** a developer adds or modifies a named query in any `db/query.<engine>.sql` file
- **THEN** they SHALL run `sqlc generate` to regenerate the per-engine packages
- **AND** they SHALL run `go generate ./pkg/database` to regenerate the hand-written `Querier` superset

### Requirement: All SQL column names are quoted

The system SHALL require every column name in every SQL query (`SELECT`, `INSERT`, `UPDATE`, `DELETE`) and every column name in every `CREATE TABLE` statement to be wrapped in double quotes — e.g. `"id"`, `"hash"`, `"url"`, `"created_at"`. This is a blanket rule applying to all three query files and all schema/migration files; it is not limited to reserved words or ambiguous identifiers.

#### Scenario: Query file authoring

- **WHEN** a developer writes or edits a query in any `db/query.<engine>.sql` file
- **THEN** every column name in the query SHALL be quoted with double quotes

#### Scenario: Migration file authoring

- **WHEN** a developer writes a migration file under `db/migrations/<engine>/`
- **THEN** every column name in every `CREATE TABLE`, `ALTER TABLE`, and DML statement SHALL be quoted with double quotes

### Requirement: dbmate manages versioned migrations

The system SHALL manage database schema migrations with dbmate. Migration files SHALL live under `db/migrations/{sqlite,postgres,mysql}/`. A wrapper binary (`nix/dbmate-wrapper/`) SHALL auto-select the per-engine migrations directory from the URL scheme so operators do not pass `--migrations-dir` explicitly.

#### Scenario: dbmate up against SQLite

- **WHEN** an operator runs `dbmate --url sqlite:/path/to/db up`
- **THEN** the wrapper SHALL invoke the real dbmate with the `db/migrations/sqlite/` directory selected automatically

#### Scenario: dbmate up against PostgreSQL or MySQL

- **WHEN** an operator runs `dbmate --url postgresql://...` or `dbmate --url mysql://...` for any subcommand
- **THEN** the wrapper SHALL select `db/migrations/postgres/` or `db/migrations/mysql/` respectively

### Requirement: dbmate `new` is the only way to create migration files

The system SHALL require developers to create new migration files via `dbmate new <name>` (or the `/migrate-new` skill, which wraps it for all engines). Manual copying of existing migration files is forbidden because it produces incorrect timestamps and breaks chronological ordering.

#### Scenario: Creating a migration

- **WHEN** a developer needs a new schema change
- **THEN** they SHALL run `dbmate new <descriptive_name>` (or invoke `/migrate-new`) so a correctly-timestamped file appears in each engine's migrations directory

### Requirement: Applying migrations regenerates the schema snapshots

The system SHALL regenerate `db/schema/sqlite.sql`, `db/schema/postgres.sql`, and `db/schema/mysql.sql` whenever migrations are applied. These snapshots SHALL reflect the post-migration state and are checked into the repository as documentation.

#### Scenario: After dbmate up

- **WHEN** dbmate applies one or more migrations against any engine
- **THEN** the corresponding `db/schema/<engine>.sql` SHALL be regenerated to reflect the new state

### Requirement: `config` table schema

The system SHALL maintain a `config` table for arbitrary key/value configuration (notably the secret signing key) with the following column shape:

```sql
CREATE TABLE "config" (
    "id"         INTEGER PRIMARY KEY AUTOINCREMENT,
    "key"        TEXT NOT NULL UNIQUE,
    "value"      TEXT NOT NULL,
    "created_at" TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL,
    "updated_at" TIMESTAMP
);
```

#### Scenario: Inserting a config value

- **WHEN** application code inserts `("secret_key", <value>)` into `config`
- **THEN** the row SHALL be created with `created_at` defaulted to the current timestamp and `updated_at` left NULL

#### Scenario: Key uniqueness

- **WHEN** an insert with an existing `key` is attempted
- **THEN** the database SHALL reject the insert with a uniqueness violation

### Requirement: `narinfos` table schema

The system SHALL maintain a `narinfos` table with one row per cached narinfo, storing the narinfo fields denormalized inline for fast lookup. The `url`, `compression`, and related fields MAY be `NULL` for narinfo stubs whose full metadata has not yet been populated.

```sql
CREATE TABLE "narinfos" (
    "id"               INTEGER PRIMARY KEY AUTOINCREMENT,
    "hash"             TEXT NOT NULL UNIQUE,
    "store_path"       TEXT,
    "url"              TEXT,
    "compression"      TEXT,
    "file_hash"        TEXT,
    "file_size"        BIGINT CHECK ("file_size" >= 0),
    "nar_hash"         TEXT,
    "nar_size"         BIGINT CHECK ("nar_size" >= 0),
    "deriver"          TEXT,
    "system"           TEXT,
    "ca"               TEXT,
    "created_at"       TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL,
    "updated_at"       TIMESTAMP,
    "last_accessed_at" TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_narinfos_last_accessed_at ON "narinfos" ("last_accessed_at");
```

#### Scenario: Inserting a complete narinfo

- **WHEN** application code inserts a narinfo with all fields populated
- **THEN** the row SHALL be created and `last_accessed_at` SHALL default to the current timestamp

#### Scenario: Stub narinfo

- **WHEN** a narinfo row exists with only `hash` populated and the rest NULL
- **THEN** the database SHALL accept the row (no NOT NULL constraint on the optional fields)

#### Scenario: CHECK violation on negative file_size

- **WHEN** an insert attempts `file_size = -1`
- **THEN** the database SHALL reject the row via the CHECK constraint

### Requirement: `narinfo_references` table schema

The system SHALL maintain a `narinfo_references` table that stores the `References` field of each narinfo (one row per referenced store path). The table SHALL have a composite primary key on `(narinfo_id, reference)` and a FOREIGN KEY on `narinfo_id` with `ON DELETE CASCADE`.

```sql
CREATE TABLE "narinfo_references" (
    "narinfo_id" BIGINT NOT NULL REFERENCES "narinfos" ("id") ON DELETE CASCADE,
    "reference"  TEXT NOT NULL,
    PRIMARY KEY ("narinfo_id", "reference")
);
CREATE INDEX idx_narinfo_references_reference ON "narinfo_references" ("reference");
```

#### Scenario: Reverse lookup by reference

- **WHEN** application code queries for all narinfos that reference a given store path
- **THEN** the secondary index `idx_narinfo_references_reference` SHALL be used

#### Scenario: Cascade delete

- **WHEN** a `narinfos` row is deleted
- **THEN** all corresponding `narinfo_references` rows SHALL be removed automatically

### Requirement: `narinfo_signatures` table schema

The system SHALL maintain a `narinfo_signatures` table that stores the `Sig` lines of each narinfo (one row per signature). The table SHALL have a composite primary key on `(narinfo_id, signature)` and a FOREIGN KEY on `narinfo_id` with `ON DELETE CASCADE`.

```sql
CREATE TABLE "narinfo_signatures" (
    "narinfo_id" BIGINT NOT NULL REFERENCES "narinfos" ("id") ON DELETE CASCADE,
    "signature"  TEXT NOT NULL,
    PRIMARY KEY ("narinfo_id", "signature")
);
CREATE INDEX idx_narinfo_signatures_signature ON "narinfo_signatures" ("signature");
```

#### Scenario: Reverse lookup by signature

- **WHEN** application code queries for all narinfos carrying a specific signature
- **THEN** the secondary index `idx_narinfo_signatures_signature` SHALL be used

#### Scenario: Cascade delete

- **WHEN** a `narinfos` row is deleted
- **THEN** all corresponding `narinfo_signatures` rows SHALL be removed automatically

### Requirement: `nar_files` table schema

The system SHALL maintain a `nar_files` table with one row per unique `(hash, compression, query)` combination. The table SHALL track both whole-file NARs and CDC-chunked NARs.

```sql
CREATE TABLE "nar_files" (
    "id"                  INTEGER PRIMARY KEY AUTOINCREMENT,
    "hash"                TEXT NOT NULL,
    "compression"         TEXT NOT NULL DEFAULT '',
    "file_size"           INTEGER NOT NULL,
    "query"               TEXT NOT NULL DEFAULT '',
    "total_chunks"        BIGINT NOT NULL DEFAULT 0,
    "chunking_started_at" TIMESTAMP NULL,
    "verified_at"         TIMESTAMP,
    "created_at"          TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL,
    "updated_at"          TIMESTAMP,
    "last_accessed_at"    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE ("hash", "compression", "query")
);
CREATE INDEX idx_nar_files_last_accessed_at ON "nar_files" ("last_accessed_at");
```

#### Scenario: Unique combination

- **WHEN** an insert attempts a `(hash, compression, query)` triple that already exists
- **THEN** the database SHALL reject the insert via the UNIQUE constraint

#### Scenario: LRU index

- **WHEN** application code queries for the least-recently-accessed NAR files
- **THEN** the secondary index `idx_nar_files_last_accessed_at` SHALL be used

### Requirement: CDC state machine on `nar_files`

The system SHALL interpret the `total_chunks` and `chunking_started_at` columns of `nar_files` as a state machine:

| `total_chunks` | `chunking_started_at` | Meaning |
|---|---|---|
| `0` | `NULL` | Not yet chunked; whole-file NAR in storage |
| `0` | NOT NULL, age < 1h | Chunking in progress; lock is fresh |
| `0` | NOT NULL, age ≥ 1h | Stale lock; cleanup and restart chunking |
| `> 0` | any | Fully chunked; whole-file NAR deleted |

#### Scenario: Reading the not-chunked state

- **WHEN** a row has `total_chunks=0` and `chunking_started_at IS NULL`
- **THEN** the storage layer SHALL treat the NAR as a whole file present in object storage

#### Scenario: Detecting a stale chunking lock

- **WHEN** a row has `total_chunks=0` and `chunking_started_at` older than 1 hour
- **THEN** the cleanup path SHALL claim the row, reset `chunking_started_at`, and restart chunking

### Requirement: `narinfo_nar_files` join table schema

The system SHALL maintain a `narinfo_nar_files` many-to-many join table allowing multiple narinfos to reference the same physical NAR file. Both foreign keys SHALL cascade on delete.

```sql
CREATE TABLE "narinfo_nar_files" (
    "narinfo_id"  INTEGER NOT NULL REFERENCES "narinfos" ("id") ON DELETE CASCADE,
    "nar_file_id" INTEGER NOT NULL REFERENCES "nar_files" ("id") ON DELETE CASCADE,
    PRIMARY KEY ("narinfo_id", "nar_file_id")
);
CREATE INDEX idx_narinfo_nar_files_narinfo_id  ON "narinfo_nar_files" ("narinfo_id");
CREATE INDEX idx_narinfo_nar_files_nar_file_id ON "narinfo_nar_files" ("nar_file_id");
```

#### Scenario: Looking up NAR files for a narinfo

- **WHEN** application code joins narinfos to their NAR files
- **THEN** the secondary index `idx_narinfo_nar_files_narinfo_id` SHALL be used

#### Scenario: Reverse lookup

- **WHEN** application code finds all narinfos referencing a NAR file
- **THEN** the secondary index `idx_narinfo_nar_files_nar_file_id` SHALL be used

### Requirement: `chunks` table schema

The system SHALL maintain a `chunks` table with one row per unique chunk content hash. Chunks SHALL be zstd-compressed on disk, with `compressed_size` tracking the on-disk byte count.

```sql
CREATE TABLE "chunks" (
    "id"              INTEGER PRIMARY KEY AUTOINCREMENT,
    "hash"            TEXT NOT NULL UNIQUE,
    "size"            INTEGER NOT NULL CHECK ("size" >= 0),
    "compressed_size" INTEGER NOT NULL DEFAULT 0 CHECK ("compressed_size" >= 0),
    "created_at"      TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL,
    "updated_at"      TIMESTAMP
);
```

#### Scenario: Hash format

- **WHEN** application code inserts a chunk
- **THEN** `hash` SHALL be the Nix base32 representation of the chunk's content, 52 characters long

#### Scenario: CHECK violations

- **WHEN** an insert attempts a negative `size` or `compressed_size`
- **THEN** the database SHALL reject the row via CHECK constraint

### Requirement: `nar_file_chunks` table schema

The system SHALL maintain a `nar_file_chunks` table storing the ordered sequence of chunks composing a chunked NAR file. The primary key SHALL be `(nar_file_id, chunk_index)`; both foreign keys SHALL cascade on delete.

```sql
CREATE TABLE "nar_file_chunks" (
    "nar_file_id"  INTEGER NOT NULL REFERENCES "nar_files" ("id") ON DELETE CASCADE,
    "chunk_id"     INTEGER NOT NULL REFERENCES "chunks" ("id") ON DELETE CASCADE,
    "chunk_index"  INTEGER NOT NULL,
    PRIMARY KEY ("nar_file_id", "chunk_index")
);
CREATE INDEX idx_nar_file_chunks_chunk_id ON "nar_file_chunks" ("chunk_id");
```

#### Scenario: Iterating a chunked NAR

- **WHEN** application code reconstructs a chunked NAR
- **THEN** it SHALL query `nar_file_chunks` ordered by `chunk_index`

#### Scenario: Reverse lookup by chunk

- **WHEN** application code asks which NAR files use a chunk
- **THEN** the secondary index `idx_nar_file_chunks_chunk_id` SHALL be used

### Requirement: All foreign keys use ON DELETE CASCADE

The system SHALL declare `ON DELETE CASCADE` on every foreign key in the schema. Removing a `narinfos` or `nar_files` row SHALL automatically remove all dependent child rows in `narinfo_references`, `narinfo_signatures`, `narinfo_nar_files`, and `nar_file_chunks`.

#### Scenario: Deleting a narinfo

- **WHEN** a `narinfos` row is deleted
- **THEN** all `narinfo_references`, `narinfo_signatures`, and `narinfo_nar_files` rows referencing that narinfo SHALL be deleted automatically

#### Scenario: Deleting a nar_file

- **WHEN** a `nar_files` row is deleted
- **THEN** all `narinfo_nar_files` and `nar_file_chunks` rows referencing that nar_file SHALL be deleted automatically

### Requirement: `database.Querier` exposes config CRUD

The system SHALL expose the following methods on `database.Querier`:

- `GetConfigByKey(ctx, key) (Config, error)`
- `SetConfig(ctx, key, value) error`
- `CreateConfig(ctx, key, value) (Config, error)`

#### Scenario: Fetching a missing key

- **WHEN** `GetConfigByKey` is called with a key that does not exist
- **THEN** the method SHALL return a `database.ErrNotFound` error

### Requirement: `database.Querier` exposes narinfo CRUD

The system SHALL expose the following methods on `database.Querier`:

- `GetNarInfoByHash(ctx, hash) (NarInfo, error)`
- `GetNarInfoHashByNarURL(ctx, url) (string, error)`
- `CreateNarInfo(ctx, CreateNarInfoParams) (NarInfo, error)`
- `UpdateNarInfoLastAccessedAt(ctx, hash) error`
- `GetLeastUsedNarInfos(ctx, limit) ([]NarInfo, error)`
- `DeleteNarInfoByHash(ctx, hash) error`
- `GetMigratedNarInfoHashes(ctx) ([]string, error)`

#### Scenario: GetMigratedNarInfoHashes returns rows whose url is populated

- **WHEN** `GetMigratedNarInfoHashes` is called
- **THEN** the result SHALL contain only the hashes of narinfos whose `url` column is non-NULL (i.e. fully migrated from the storage backend to the database)

### Requirement: `database.Querier` exposes nar_file CRUD

The system SHALL expose the following methods on `database.Querier`:

- `GetNarFileByHashAndCompressionAndQuery(ctx, hash, compression, query) (NarFile, error)`
- `GetNarFileByNarInfoID(ctx, narInfoID) (NarFile, error)`
- `CreateNarFile(ctx, CreateNarFileParams) (NarFile, error)`
- `UpdateNarFileLastAccessedAt(ctx, id) error`
- `GetLeastUsedNarFiles(ctx, limit) ([]NarFile, error)`
- `DeleteNarFileByID(ctx, id) error`
- `GetOrphanedNarFiles(ctx) ([]NarFile, error)`

#### Scenario: GetOrphanedNarFiles returns nar_files with no narinfo

- **WHEN** `GetOrphanedNarFiles` is called
- **THEN** the result SHALL include only nar_file rows for which no `narinfo_nar_files` linkage exists

### Requirement: `database.Querier` exposes chunk CRUD

The system SHALL expose the following methods on `database.Querier`:

- `CreateChunk(ctx, CreateChunkParams) (Chunk, error)`
- `GetChunkByHash(ctx, hash) (Chunk, error)`
- `CreateNarFileChunk(ctx, narFileID, chunkID, chunkIndex) error`
- `GetChunksByNarFileID(ctx, narFileID) ([]Chunk, error)`
- `GetOrphanedChunks(ctx) ([]Chunk, error)`
- `DeleteChunkByID(ctx, id) error`
- `DeleteNarFileChunksByNarFileID(ctx, narFileID) error`

#### Scenario: GetChunksByNarFileID preserves order

- **WHEN** `GetChunksByNarFileID` is called for a chunked NAR
- **THEN** the result SHALL be ordered by `nar_file_chunks.chunk_index` ascending

### Requirement: `database.Querier` supports transactions

The system SHALL expose `WithTransaction(ctx, fn func(Querier) error) error` on `database.Querier`. The function SHALL begin a database transaction, invoke `fn` with a `Querier` bound to that transaction, and commit if `fn` returns nil or roll back if `fn` returns an error.

#### Scenario: Successful transaction

- **WHEN** `fn` returns nil after performing one or more queries
- **THEN** all queries SHALL be committed atomically

#### Scenario: Failed transaction rolls back

- **WHEN** `fn` returns a non-nil error after performing one or more queries
- **THEN** every query SHALL be rolled back and no row SHALL persist
