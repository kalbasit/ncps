## Why

Production clients hit `error: file 'nar/<hash>.nar' does not exist in binary cache` (and `Lzma: No progress is possible`) that abort `nix` builds, even though the NAR bytes are present locally. The cause is a **narinfo↔nar_file compression desync**: a NAR whose narinfo advertises `Compression: none` is physically stored whole-file as `.nar.xz`, but the serve path only ever transparently decompresses a stored `.nar.zst` for a `none` request — never `.nar.xz` — so it reports the NAR absent and 404s / re-downloads. DB inspection of the live cache found ~259 drifted narinfo/nar_file pairs; all client-reported failing hashes were `narinfo=none` / `stored=xz`. The drift is generated by CDC (which normalizes narinfos to `none`) and left as residue after CDC is disabled and NARs re-download at the upstream's compression, so it is CDC-state-independent and was independently reproduced on a steady-state CDC deployment. Tracking issue: #1392 (distinct from the closed #1329 opaque-nar-url parsing bug).

## What Changes

- The NAR serve path resolves a whole-file NAR by hash and serves a `Compression: none` request by transparently decompressing **any** stored whole-file compression, not just `.nar.zst`. This adds `.nar.xz` (and any other supported codec) to the existing transparent-decompress convention, so a locally-present NAR is served from disk instead of 404/re-downloaded.
- The existence check used to gate serving (`statNarInStore`) is generalized the same way, so a `none` request reports present when only a compressed whole file (e.g. `.nar.xz`) is on disk.
- When a `none` request is served by decompressing a stored compressed whole file, LRU tracking touches the nar_file row matching the bytes actually served, rather than synthesizing a spurious `none` row.
- A regression test reproduces the production failure: store a NAR as a whole-file `.nar.xz`, request it uncompressed with no upstream, and assert it is served decompressed (was `not found`).

Out of scope (deferred to follow-ups tracked in #1392): the inverse direction (narinfo advertises `xz`/`zstd` while the NAR is stored uncompressed or as chunks, which needs serve-time **re-compression**); a one-shot data repair to reconcile existing drifted pairs; and a write-path invariant keeping a narinfo's advertised compression equal to a servable representation.

## Capabilities

### New Capabilities
- `nar-serving-compression-fallback`: Serving a `Compression: none` NAR request by transparently decompressing whichever supported whole-file compression is actually stored (zstd, xz, …), so a locally-present NAR is never reported absent due to a narinfo↔nar_file compression mismatch.

## Impact

- `pkg/cache/cache.go`: `getNarFromStore` (serve + transparent decompress + LRU touch), `statNarInStore` (existence check), new `wholeFileServeCompressions()` helper.
- New test `pkg/cache/narinfo_nar_compression_desync_serve_internal_test.go` (`TestServeNoneRequestFromXzWholeFile`).
- Behavior-only change; no schema, migration, config, or API surface changes. Reduces unnecessary upstream re-downloads for desynced NARs.
